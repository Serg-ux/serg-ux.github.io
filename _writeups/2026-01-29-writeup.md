## Reconnaissance

The initial step involved verifying host availability using ICMP.

```bash
ping -c <packet number> <IP>
```

Once the host was confirmed to be alive, a scan targeting common ports was performed.

```bash
nmap --open -Pn 10.129.61.130 --stats-every=5s
```

This revealed two open ports:
- 22 (SSH)
- 80 (HTTP)

A more detailed scan was then conducted to identify service versions and retrieve default scripts output.

```bash
nmap -p 22,80 -sC -sV -Pn 10.129.61.130 --stats-every=5s -oN scan_ports.txt
```

This provided valuable information about the services running on the target.

![nmap version scan results for the open ports](/assets/images/writeup/Pasted-image-20260129195802.png)
*Service details returned for ports 22 and 80*

## Enumeration

Given the presence of a web server, the focus shifted to web application analysis. Burp Suite was used to intercept and inspect HTTP traffic, enabling a deeper understanding of the application's behavior.

Initial enumeration included reviewing page functionality, identifying technologies and versions, and searching for exposed users or endpoints.

The `robots.txt` file revealed a disallowed directory.

![robots.txt revealing a disallowed directory](/assets/images/writeup/Pasted-image-20260129200131.png)
*Disallowed entry found in robots.txt*

The `/writeup/` directory appeared to host Hack The Box writeups, suggesting user-generated content. Additionally, references to a "cool theme" hinted at potential CMS-related components.

Attempts to enumerate subdomains and directories resulted in blocking, likely due to defensive mechanisms.

![directory enumeration being blocked](/assets/images/writeup/Pasted-image-20260129201656.png)
*Enumeration attempt being blocked by the target*

![subdomain enumeration also being blocked](/assets/images/writeup/Pasted-image-20260129201710.png)
*Subdomain enumeration attempt blocked in the same way*

Further inspection of HTTP requests revealed a session cookie.

![session cookie observed in the HTTP request](/assets/images/writeup/Pasted-image-20260129201836.png)
*Session cookie returned by the application*

The cookie name `CMSSESSID9d372ef93962` strongly suggested the use of a CMS.

Reviewing the page source confirmed the technology in use.

![page source confirming the CMS in use](/assets/images/writeup/Pasted-image-20260129201813.png)

The application was running **CMS Made Simple (2004–2019)**.

## Exploitation

### CMS Vulnerability Exploitation

Research into known vulnerabilities affecting the identified CMS version led to the discovery of a public exploit.

![public exploit identified for CMS Made Simple](/assets/images/writeup/Pasted-image-20260129201739.png)
*Public exploit located for the identified CMS version*

Due to compatibility issues, the exploit was modified to run under Python 3. After execution, it successfully extracted a password hash and associated credentials.

![password hash and credentials extracted by the exploit](/assets/images/writeup/Pasted-image-20260129204832.png)
*Credentials and password hash extracted via the CMS exploit*

Since no administrative panel or direct login vector was available, and Pass-the-Hash was not applicable, the next step was offline password cracking.

### Hash Cracking

The extracted hash required formatting as `HASH:SALT`. Given that it was an MD5-based hash, Hashcat was used.

```bash
hashcat -a0 -m 20 hash /usr/share/wordlists/rockyou.txt
```

After successfully cracking the password, valid credentials were obtained. These credentials allowed SSH access to the system.

![SSH access obtained with the cracked credentials](/assets/images/writeup/Pasted-image-20260129205335.png)
*SSH session established using the cracked credentials*

## Post-Exploitation

After gaining initial access as a low-privileged user, enumeration for privilege escalation vectors began.

### System Enumeration

Sudo permissions were checked.

```bash
sudo -l
```

User enumeration was performed.

```bash
cat /etc/passwd
```

Command history was reviewed.

```bash
cat .bash_history
```

Scheduled tasks were inspected.

```bash
crontab -l
```

Network services were enumerated.

```bash
netstat -ano
```

A MySQL service was identified, but access attempts failed.

![MySQL access attempt failing](/assets/images/writeup/Pasted-image-20260129210637.png)

Processes running as root were inspected.

```bash
ps aux | grep "root"
```

### Group-Based Privilege Analysis

Checking group memberships revealed an unusual privilege level.

```bash
id
```

![id command output showing group memberships](/assets/images/writeup/Pasted-image-20260129210849.png)
*Group memberships of the current user*

The user belonged to the `staff` group. On Debian-based systems, this group can modify directories such as `/usr/local`, which may lead to privilege escalation if abused.

Permissions on relevant directories were verified.

```bash
ls -ld /usr/local/bin/ /usr/local/sbin/
```

## Privilege Escalation

### Path Hijacking via Writable Binary

To identify exploitable binaries, [`pspy`](https://github.com/DominicBreuker/pspy) was used to monitor processes executed by root. `pspy` is a process-monitoring tool that does not require root privileges, making it useful for observing scheduled or triggered root processes from a low-privileged shell.

While initial observations did not reveal much, reconnecting via SSH triggered execution of a binary.

![pspy showing a binary executed by root on SSH reconnect](/assets/images/writeup/Pasted-image-20260129212928.png)
*Root-owned process triggered upon SSH reconnection*

Further analysis using [GTFOBins](https://gtfobins.org/gtfobins/run-parts/) indicated that `run-parts` could be abused. The `run-parts` binary executes all scripts located in `/etc/update-motd.d` as root. However, direct modification of this directory was not possible due to insufficient permissions.

![permission denied when attempting to modify /etc/update-motd.d](/assets/images/writeup/Pasted-image-20260129214454.png)
*Insufficient permissions to modify /etc/update-motd.d directly*

### Exploitation Strategy

Since the user had write permissions in `/usr/local/bin` (due to `staff` group membership), a path hijacking attack was performed by replacing the `run-parts` binary with a malicious script.

> Because `/usr/local/bin` appears earlier in `$PATH` than `/usr/bin`, a script placed there with the same name as a system binary will be executed in its place — even by processes running with higher privileges, as long as they invoke the binary without an absolute path.

The payload was designed to set the SUID bit on `/bin/bash`.

```bash
echo -e '#!/bin/bash\n\nchmod u+s /bin/bash' > /usr/local/bin/run-parts; chmod +x /usr/local/bin/run-parts
```

Upon re-establishing the SSH session, the modified binary was executed automatically, resulting in privilege escalation.

![SUID bit successfully set on /bin/bash](/assets/images/writeup/Pasted-image-20260129215204.png)
*Confirmation that /bin/bash now has the SUID bit set*

A root shell was obtained via the SUID-enabled bash binary, completing the attack chain.
