## Host Configuration

The target IP is added to the local `/etc/hosts` file to ensure proper hostname resolution before beginning the engagement.

```bash
nano /etc/hosts
```

## Network Scan

A full port scan is run against the target to identify all open services.

```bash
nmap -Pn -sV -sC -p- --open 10.10.11.243 -oN scan.txt
```

|Flag|Purpose|
|---|---|
|`-Pn`|Disables host discovery — treats the host as always online|
|`-sV`|Detects service versions|
|`-sC`|Runs default NSE scripts to gather additional service information|
|`-p-`|Scans all 65535 ports|
|`--open`|Only displays open ports|
|`-oN scan.txt`|Saves output to a file|

The scan reveals that an **Apache ActiveMQ** service is exposed — a message broker that becomes the primary attack vector.

## Exploitation — CVE-2023-46604

The version of Apache ActiveMQ running on the target is vulnerable to **CVE-2023-46604**, a critical unauthenticated **Remote Code Execution** vulnerability. The flaw exists in the OpenWire protocol deserialisation handler: a specially crafted `ExceptionResponse` packet can trick the broker into loading a remote ClassInfo object, executing arbitrary code on the server without any authentication required.

> CVE-2023-46604 carries a CVSS score of 10.0 — it requires no credentials and gives immediate RCE, making it one of the most severe ActiveMQ vulnerabilities to date.

## Privilege Escalation — nginx Misconfiguration

### Checking Sudo Permissions

After gaining a foothold via the RCE, sudo permissions are enumerated to identify potential privilege escalation paths.

```bash
sudo -l
```

The output reveals that the current user can run `nginx` as root without a password — a significant misconfiguration that allows an arbitrary nginx config file to be loaded with root privileges.

### Crafting a Malicious nginx Config

The default nginx config is copied to `/tmp` as a starting point.

```bash
cp /etc/nginx/nginx.conf /tmp/pwnd.conf
```

The config is then modified to define a server that serves the filesystem root (`/`) as its document root, listening on a local port. This allows direct file reads as root through HTTP requests.

### Starting nginx with the Custom Config

The modified config is loaded using sudo.

```bash
sudo nginx -c /tmp/pwnd.conf
```

Since nginx starts as root, it has read access to all files on the system — including those owned by root.

### Reading the Root Flag

With the server running, the `root.txt` flag is retrieved directly via a curl request to the local port.

```bash
curl 127.0.0.1:9001/root.txt
```

> This technique abuses the fact that nginx, when run as root, inherits root-level file permissions. By pointing the document root at `/`, any file on the system can be read over HTTP — no shell needed.
