## Reconnaissance

The engagement began with a full TCP port scan to identify exposed services and gather version information.

```bash
nmap 10.10.11.13 -Pn -sC -sV -p- --open --stats-every=5s -oA scan
```

The scan revealed three main services: SSH on port 22, an HTTP service on port 80 served by nginx, and a Nagios NSCA service on port 8000. The web application did not resolve at first, so the domain was added to the local `/etc/hosts` file.

To fingerprint the application's technologies, WhatWeb was run against the target.

```bash
whatweb http://runner.htb/
```

![To identify the technologies used by the application, running WhatWeb against the target](/assets/images/runner/40-1.png)
*To identify the technologies used by the application, running WhatWeb against the target*

## Enumeration / Analysis

With the web application accessible, directory brute-forcing was performed to find interesting endpoints, login pages, or upload functionality.

```bash
gobuster dir -u http://runner.htb/ -w /usr/share/wordlists/dirb/big.txt
```

This revealed an `/assets` directory but no immediately exploitable functionality. Virtual-host discovery with ffuf was attempted next, though it produced no meaningful results.

```bash
ffuf -u http://runner.htb/ -H "Host: FUZZ.runner.htb" -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt -fs 3058
```

Switching to Gobuster’s vhost mode identified several virtual hosts. One stood out because it exposed a login interface.

```bash
gobuster vhost --append-domain -u http://runner.htb/ -w /usr/share/seclists/Discovery/DNS/namelist.txt --random-agent -t 600
```

![Switching to Gobuster’s vhost mode, which successfully identified several virtual hosts](/assets/images/runner/40-2.png)
*Switching to Gobuster’s vhost mode, which successfully identified several virtual hosts*

![Switching to Gobuster’s vhost mode, which successfully identified several virtual hosts](/assets/images/runner/40-3.png)
*Switching to Gobuster’s vhost mode, which successfully identified several virtual hosts*

This virtual host was added to `/etc/hosts`. Visiting it redirected to a TeamCity login page, revealing a JetBrains TeamCity CI/CD server. The version was disclosed directly on the login page, which helped narrow down potential vulnerabilities.

## Exploitation

To find known vulnerabilities affecting TeamCity, Nuclei was run with the appropriate tags. The scan flagged a critical authentication-bypass vulnerability.

```bash
nuclei -tags teamcity -u teamcity.runner.htb
```

Further research confirmed the issue as CVE-2024-27198, an authentication bypass through an alternative path. A public exploit makes it possible to create a new administrative user and gain access to the TeamCity dashboard.

```python
python3 CVE-2024-27198.py -t http://teamcity.runner.htb/login.html -u sergux -p sergux2
```

![Further research confirmed the issue as CVE-2024-27198, which allows authentication bypass through an alternative vulnerable path](/assets/images/runner/40-4.png)

Once authenticated as an administrator, the existing users were enumerated while looking for ways to run arbitrary commands on the underlying system. This led to another known vulnerability, CVE-2023-42793, which allows remote code execution through TeamCity.

```bash
bash CVE-2023-42793_rce.sh teamcity.runner.htb 80 "id"
```

After confirming command execution, the exploit was modified to trigger a reverse shell. The payload downloaded and ran a Bash reverse-shell script hosted on the attacking machine, providing a more stable interactive session.

![After confirming command execution, modifying the exploit to trigger a reverse shell](/assets/images/runner/40-9.png)

![After confirming command execution, modifying the exploit to trigger a reverse shell](/assets/images/runner/40-10.png)

Post-exploitation enumeration uncovered a private SSH key stored at the following location:

```bash
/data/teamcity_server/datadir/config/projects/AllProjects/pluginData/ssh_keys/id_rsa
```

Testing this key against the previously identified users allowed authentication over SSH. Where permission issues arose, setting the file mode to `600` resolved them.

![Using this key and testing it against previously identified users, authenticating via SSH](/assets/images/runner/40-11.png)
*Using this key and testing it against previously identified users, authenticating via SSH*

## Privilege Escalation

With SSH access established, `linpeas.sh` was uploaded to continue local enumeration. It revealed an additional internal subdomain.

![With SSH access established, uploading to continue local enumeration](/assets/images/runner/40-12.png)
*With SSH access established, uploading to continue local enumeration*

After adding the new subdomain to `/etc/hosts`, visiting it returned a login page, which further inspection identified as a Portainer instance.

![The new subdomain resolving to a login page](/assets/images/runner/40-13.png)
*The new subdomain resolving to a login page*

From this interface, it is possible to generate a full database backup, which contained password hashes for all users.

![Generating a full database backup from Portainer](/assets/images/runner/40-14.png)
*Generating a full database backup from Portainer*

![Password hashes extracted from the Portainer backup](/assets/images/runner/40-15.png)
*Password hashes extracted from the Portainer backup*

Cracking the hashes with Hashcat recovered valid credentials for the user `matthew`.

```bash
hashcat -m 3200 -a 0 hash /usr/share/wordlists/rockyou.txt --show
```

The credentials **matthew:piper123** granted access to the Portainer dashboard. Because Portainer manages Docker containers, this opened a clear path to privilege escalation.

![Using the credentials matthew:piper123, logging into the Portainer dashboard](/assets/images/runner/40-16.png)
*Using the credentials matthew:piper123, logging into the Portainer dashboard*

A new Docker volume was created mapped to the root filesystem, and a container was deployed from the TeamCity image. The container was configured with an interactive TTY and run as root, with the host's root volume mounted inside.

![Deploying a TeamCity container with the host root filesystem mounted](/assets/images/runner/40-17.png)
*Deploying a TeamCity container with the host root filesystem mounted*

*Deploying a TeamCity container with the host root filesystem mounted*

*Deploying a TeamCity container with the host root filesystem mounted*

![Deploying a TeamCity container with the host root filesystem mounted](/assets/images/runner/40-20.png)

![Deploying a TeamCity container with the host root filesystem mounted](/assets/images/runner/40-21.png)

Connecting to the container provided full root access to the host filesystem, allowing the root flag to be retrieved and completing the compromise.
